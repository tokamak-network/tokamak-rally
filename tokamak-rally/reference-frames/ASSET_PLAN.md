# Tokamak Rally v3 에셋 리뉴얼 계획서

## 레퍼런스 분석 요약

### Thrash Rally 프레임에서 배운 것

| 요소 | Thrash Rally | 현재 우리 게임 | 개선 방향 |
|------|-------------|--------------|----------|
| **차량 크기** | 도로 너비의 ~25-30% | 스케일 0.85로 과대 | 도로 너비 대비 적절한 비율로 축소 |
| **도로변 오브젝트 밀도** | 매우 높음 — 도로 양쪽이 거의 빈틈없이 채워짐 | 간헐적 배치, 빈 공간 많음 | 밀도 2-3배 증가 |
| **배경 타일** | 타일이 아니라 전체 환경이 "가득 채워진" 느낌 | AI 생성 타일의 투명 배경 문제 | 완전 불투명 타일 + 프로그래매틱 디테일 |
| **색상 팔레트** | 존별 명확한 색온도 구분 (사막=따뜻한 황갈, 숲=깊은 초록, 도시=차가운 회색) | 존 간 색감 구분 약함 | 존별 엄격한 팔레트 제한 |
| **배리어/경계** | 빨강-흰색 스트라이프 컬링 + 나무 울타리 등 환경에 맞는 소재 | 프로그래매틱 곡선, 비현실적 | 존별 고유 배리어 스프라이트 |
| **오브젝트 스타일** | 깔끔한 픽셀아트, 실루엣이 명확 | AI 생성 = 헤일로/배경색 문제 | Canvas 기반 프로그래매틱 픽셀아트 |

### 핵심 인사이트
1. **Thrash Rally의 비결 = 빈틈없는 밀도** — 도로 양쪽이 환경 오브젝트로 꽉 차있어 빈 배경이 노출되지 않음
2. **보색 대비** — 차량(파랑)이 항상 배경(갈색/초록)과 보색으로 즉시 식별 가능
3. **환경 전환 = 팔레트 전환** — 새 스테이지 진입 시 색상 체계가 완전히 바뀜
4. **단순하지만 명확한 실루엣** — 각 오브젝트가 16-24px 수준에서도 즉시 인식 가능

## 에셋 제작 전략: Canvas 프로그래매틱 픽셀아트

### 왜 AI 이미지 생성 대신 프로그래매틱?
- AI 생성 PNG의 투명도 문제 근본 해결
- 정확한 팔레트 컨트롤 가능
- 존별 색감 통일 보장
- 크기/비율 정밀 제어
- 런타임에서 팔레트 스왑으로 전환 존 구현 용이

### 제작 방법
Node.js 스크립트로 Canvas에 픽셀 단위로 각 에셋을 그려서 PNG 저장. 
Thrash Rally 프레임의 색상을 **정확히 eyedrop**해서 팔레트 구성.

## 구현 순서

### Phase 1: 배경 타일 + 배리어 (기반)
1. 5개 존 배경 타일 재생성 (프로그래매틱, 완전 불투명)
2. 5개 배리어 스프라이트 (존별 소재: 컬링/돌벽/나무/눈벽/콘크리트)
3. RaceScene.js에서 배경 렌더링 방식 개선

### Phase 2: 도로변 오브젝트 (밀도)
4. 각 존 오브젝트 전체 재생성 (위 표 기준)
5. placeScenery() 로직 수정 — 밀도 2-3배 증가, 도로 양쪽 빈틈없이 채움
6. 전환 존 오브젝트 혼합 배치 로직

### Phase 3: 차량 + 인터랙션
7. 5대 차량 스프라이트 재생성 (탑다운 픽셀아트)
8. 차량 그림자 + 아웃라인 추가
9. 차량 스케일 재조정 (도로 너비 대비 25-30%)

### Phase 4: Sprint 특수 요소
10. Sprint 건물 밀도 대폭 증가
11. Tokamak 배너/광고판 개선
12. 관중석 + 군중 개선
13. 피니시 구역 리뉴얼

## 예상 에셋 총 수
- 배경 타일: 5개
- 배리어: 5개  
- 오브젝트: ~45개 (존당 8-10개)
- 차량: 5개
- 특수: ~5개 (피니시, 배너 등)
- **총 ~65개 PNG**

## 기술 스택
- `node-canvas` 패키지로 프로그래매틱 생성
- 각 에셋에 정확한 팔레트 + 크기 + 디더링 패턴 지정
- 생성 스크립트: `scripts/generate-assets-v3.js`
- 출력 경로: `public/assets/v3/`
