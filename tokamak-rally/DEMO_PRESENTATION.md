## 🎮 Part 1: 게임 소개 (3분)

### 컨셉
- **다카르 랠리 스타일 타임어택** — 비포장 사막/협곡/산악 코스
- **브라우저 기반** — 설치 없이 URL만 공유하면 플레이 가능
- **오픈소스** — `tokamak-network/tokamak-rally` 레포 공개

### 기술 스택
| 구분 | 기술 |
|------|------|
| 게임 엔진 | Phaser.js (브라우저) |
| 빌드 | Vite |
| 그래픽 | 100% 프로시저럴 픽셀아트 (외부 에셋 없음) |
| 사운드 | Web Audio API 프로시저럴 (엔진음, 드리프트, 유로비트 BGM) |
| 온체인 | ethers.js + MetaMask (Sepolia 테스트넷) |
| 스마트 컨트랙트 | RallyLeaderboard.sol (Sepolia 배포 완료) |

### 게임 특징
- **5개 존**: Desert → Canyon → Riverbed → Mountain → Sprint
- **5대 차량**: 각각 다른 스탯 (TOKAMAK GT는 팀카 🔵)
- **실제 드리프트 메커닉**: 방향키+스페이스 = 드리프트, 해제 시 부스트
- **결정론적 장애물**: 시드 RNG로 매번 동일한 코스 → 공정한 타임어택
- **지갑 없이 플레이 가능**: Guest 모드로 자유롭게, 기록 제출만 지갑 필요

### 시연 순서

**1) 메뉴 화면**
- 차량 선택 (TOKAMAK GT 소개 — 토카막 블루 액센트)

**2) 레이스 플레이**
- Desert 존 출발 → 비포장 먼지 이펙트 확인
- Canyon 헤어핀 → 드리프트 시연 (먼지 극대화)
- 체크포인트 통과 시 +15초
- 장애물 충돌 → 속도 페널티
- Sprint 존 → 관중 + 토카막 배너 월
- 피니시 라인 통과

**3) 기록 제출 (지갑 연동)**
- MetaMask 연결 → Sepolia 테스트넷
- Submit Record → 트랜잭션 서명
- 리더보드 확인

### 시연 포인트
- ⌨️ 조작: ↑가속 ↓브레이크 ←→조향 SPACE+방향=드리프트
- 🏁 20초 시작 → 체크포인트마다 +15초 (총 80초 예산)
- 💀 "쉽지 않습니다" — 첫 시도에 클리어하기 어려움

## 💰 Part 3: 토크노믹스 (5분)

### 솔직한 전제

> "저는 토크노믹스 전문가가 아닙니다.
> 오늘 공유하는 건 초안이고, 팀의 의견을 듣고 보강하려는 목적입니다.
> 특히 TON 생태계와의 정합성 부분은 확신이 부족합니다."

### 현재 구조 (Phase 1 — 지금)

```
플레이어 → 레이스 → 기록 제출 (Sepolia 가스비만) → 리더보드
```
- 토큰 이동 없음, 순수 타임어택 경쟁
- 이 단계에서 오늘 여러분이 테스트해볼 수 있음

### 계획 중인 구조 (Phase 2 — 토너먼트)

```
참가자 N명 × Entry Fee (TON) → Prize Pool
  ├── 90% → 우승자 (1위 50%, 2위 30%, 3위 10%)
  └── 10% → 운영자 (인프라 유지)
```

### 레퍼런스: Tokamon 컨트랙트에서 차용한 3가지 패턴

Zena가 만든 Tokamon 컨트랙트(`tokamak-network/tokamon`)를 분석하고, Rally에 필요한 3가지 패턴을 참고했습니다.

**① UUPS Upgradeable — "나중에 고칠 수 있는 컨트랙트"**

스마트 컨트랙트는 한 번 배포하면 코드 수정이 불가능합니다. 하지만 Rally는 운영하면서 조정이 반드시 필요합니다:
- 대회 주최자 / 참가자 간 **상금 배분 비율** 조정
- 대회 주최 시 **최소 TON 예치량** 변경 (무자본 주최 방지)
- Sepolia → Titan L2 **네트워크 전환** 시 로직 변경

프록시 패턴을 사용하면 데이터(유저 기록, 잔액)는 유지하면서 로직만 교체할 수 있습니다. Tokamon도 서비스 초기 규칙 변경이 잦아서 동일한 패턴을 사용 중입니다.

**② 역할 분리 — "한 키가 뚫려도 전부 털리지 않는 구조"**

하나의 관리자 키가 모든 권한을 가지면, 키 유출 시 전체 자금이 위험합니다. 상금을 다루는 만큼 권한을 역할별로 분리합니다:

| 역할 | 권한 | 할 수 없는 것 |
|------|------|-------------|
| **admin** | 컨트랙트 업그레이드, 역할 변경 | 기록 제출, 상금 배분 |
| **resultSubmitter** | 레이스 결과 체인 기록 | 상금 인출, 업그레이드 |
| **payoutManager** | 토너먼트 종료 후 상금 배분 | 결과 조작, 업그레이드 |

Tokamon은 `admin` + `claimManager` 2역할, Rally는 상금 관리의 중요성을 고려해 3역할로 더 세분화했습니다.

**③ 쿨다운 — "봇/어뷰징 방지"**

무제한 연속 참가를 허용하면 봇이 토너먼트를 점령하거나 한 명이 상금 풀을 독식할 수 있습니다. Rally에서는 **10회 연속 참가 가능 → 이후 1시간 쿨다운**으로 설계합니다. 적당한 자유도를 주되 무한 반복은 차단하는 구조입니다. Tokamon은 같은 매장 24시간 재방문 제한 + 텔레그램/디바이스 교차 검증으로 동일한 목적을 달성합니다.

### 의도적으로 뺀 것

- **Burn (소각)**: TON 소각은 네트워크 거버넌스 레벨의 결정이며, 게임 하나가 단독으로 시행할 사안이 아닙니다
- **스탬프 보너스**: Tokamon에는 있지만, Rally 토너먼트 모델과는 맞지 않아 제외했습니다

### 마이그레이션 경로

| 단계 | 네트워크 | 기능 | 비고 |
|------|----------|------|------|
| **Phase 1** (현재) | Sepolia L1 | 리더보드 | 오늘 테스트 가능 |
| Phase 2 | Sepolia | 토너먼트 mock | 테스트 TON으로 |
| Phase 3 | **Titan L2** | 프로덕션 | 실제 TON 토큰 |

## ❓ Part 4: 피드백 요청 (2분)

### 특히 듣고 싶은 의견

1. **토크노믹스 구조**
   - 90/10 배분 비율이 적절한가?
   - 대회 주최자 최소 예치량은 어느 수준이 적절한가?
   - Entry fee 방식 말고 다른 토큰 유틸리티 아이디어?
   - Titan L2에서의 가스비 구조와의 시너지?

2. **컨트랙트 설계**
   - 3역할 분리(admin/resultSubmitter/payoutManager)가 충분한지?
   - 쿨다운 10회+1시간이 적절한지?
   - 리플레이 해시만으로 결과 검증이 충분한지 vs 서버 사이드 검증?

3. **게임/생태계 핏**
   - 이런 게임이 Tokamak 생태계에 의미가 있는지?
   - 오픈소스로 풀고 커뮤니티가 운영하는 모델이 맞는지?
   - 다른 형태의 게임이나 인터랙션이 더 적합한지?
